# СтройФото — Offline-First PWA для фотоучёта строительства

Монорепозиторий для создания фотоотчётов на строительных объектах с полной поддержкой офлайн-режима.

## Архитектура

```
┌──────────────────┐       ┌──────────────────┐       ┌──────────────┐
│    apps/web      │       │    apps/api      │──────▶│   Supabase   │
│                  │       │                  │       │  PostgreSQL  │
│  React 19 SPA    │──────▶│  Fastify 5       │       └──────────────┘
│  Dexie (IDB)     │  HTTP │  Supabase Auth   │       ┌──────────────┐
│  Workbox SW      │       │                  │──────▶│ Cloudflare   │
│  Tailwind CSS 4  │       │                  │       │  R2 Storage  │
└──────────────────┘       └──────────────────┘       └──────────────┘
         │                          ▲
         │        packages/shared   │
         └────────────────────────────┘
              Zod-схемы, типы, константы
```

### Потоки данных

```
Создание отчёта (офлайн):
  Форма → Dexie reports + photos → SyncQueue → [при появлении сети] → API → Supabase PostgreSQL + Cloudflare R2

Синхронизация:
  SyncQueue обрабатывает операции в порядке: Отчёты → Фото → Финализация
  Каждая операция идемпотентна (clientId + X-Idempotency-Key)
  При 401 → автоматический refresh токена → повтор
  При 5xx → экспоненциальный backoff (5с × 2^N, max 5 мин)

Справочные данные:
  /api/reference/* → Dexie (projects, workTypes, contractors, areas)
  TTL: 24 часа, затем инкрементальный sync по updatedSince
```

### Ключевые принципы

- **Offline-first**: все данные сохраняются в IndexedDB (Dexie) мгновенно, синхронизируются при наличии сети
- **Outbox-очередь**: записи и фото попадают в очередь синхронизации, обрабатываются автоматически
- **Идемпотентный sync**: UUID генерируются на клиенте, сервер делает upsert по clientId
- **PWA**: приложение устанавливается на телефон, работает офлайн после первого визита
- **Token auto-refresh**: access token (15 мин) обновляется автоматически через refresh token (30 дней)

## Быстрый старт

```bash
# 1. Клонировать и перейти в директорию
cd stroyfoto

# 2. Создать файл переменных окружения для API
cat > apps/api/.env << 'EOF'
# Обязательные
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_SERVICE_ROLE_KEY=your-service-role-key
R2_ACCOUNT_ID=your-r2-account-id
R2_ACCESS_KEY_ID=your-r2-access-key

# R2_BUCKET_NAME=stroyfoto

# PRESIGNED_URL_EXPIRY=3600

# 3. Выполнить SQL-скрипт в Supabase SQL Editor (supabase/migrations/prod.sql)

pnpm install

pnpm dev
```

После запуска:
- **Web-приложение**: http://localhost:5173

## Карточка отчёта

Каждый отчёт содержит:
- **projectId** — идентификатор проекта/объекта
- **dateTime** — дата и время
- **workTypes[]** — типы работ (множественный выбор из справочника)
- **contractor** — подрядчик
- **ownForces** — собственные силы (опционально)
- **description** — описание
- **photos[]** — до 20 фотографий (максимум 15 МБ каждая, автосжатие до 1920px)

## Guardrails и Edge Cases

### Защитные ограничения
- **Макс. 20 фото на отчёт** — ограничение в UI и валидация при добавлении
- **Макс. 15 МБ на файл** — проверка до и после сжатия
- **Автосжатие фото** — JPEG, max 1920px по ширине, quality 0.8
- **Дедупликация** — SHA-256 хэш фото предотвращает повторную загрузку одинаковых файлов
- **Экспоненциальный backoff** — повторные попытки при 5xx: 5с, 10с, 20с... до 5 мин

### Обработка edge cases
- **Expired token при синхронизации** — автоматический refresh через `/api/auth/refresh`, повтор операции
- **Частичный сбой** — каждое фото и отчёт синхронизируются отдельно; при ошибке одного элемента остальные продолжают синхронизацию
- **Устаревший кэш справочников** — TTL 24 часа, после чего инкрементальное обновление
- **Потеря сети во время sync** — операция помечается как failed с retryable=true, повторится при восстановлении связи
- **Дублированная отправка** — идемпотентность по clientId + X-Idempotency-Key

### Управление хранилищем
- **Несинхронизированные данные НИКОГДА не удаляются автоматически**
- Кнопка «Очистить локальные копии» на странице /sync — удаляет blob-данные только для фото со статусом `synced`
- Метаданные (clientId, хэш, имя файла) сохраняются
- Перед очисткой показывается оценка освобождаемого пространства

## Ограничения офлайн-режима

1. **Первый вход** — требуется подключение к сети для авторизации
2. **Справочные данные** — кэшируются с TTL 24 часа; при первом входе без кэша формы показывают текстовые поля вместо выпадающих списков
3. **JWT access token** — 15 минут; автоматически обновляется через refresh token (30 дней). Если refresh token истёк — потребуется повторный вход
4. **Хранилище устройства** — фото хранятся как Blob в IndexedDB; при большом количестве отчётов может потребоваться очистка
5. **Конфликты** — в v1 используется last-write-wins; записи append-only по своей природе
6. **Service Worker** — регистрируется только в production build; в dev-режиме app shell не кэшируется

## Заметки для production

### HTTPS обязателен
- Service Worker регистрируется только на HTTPS (кроме localhost)
- PWA установка и Web App Manifest требуют HTTPS
- Presigned URL для Cloudflare R2 работают через HTTPS по умолчанию

### Безопасность
- `SUPABASE_SERVICE_ROLE_KEY` — хранить в секрете, не коммитить в репозиторий
- `R2_ACCESS_KEY_ID` / `R2_SECRET_ACCESS_KEY` — хранить в секрете, не коммитить в репозиторий
- Рассмотрите `navigator.storage.persist()` для предотвращения автоочистки IndexedDB браузером
- В production рекомендуется ограничить CORS (`origin: true` → конкретные домены)

### Рекомендации по деплою
- API: Docker контейнер или Node.js process manager (PM2)
- Web: статический хостинг (Nginx, Cloudflare Pages, Vercel)
- БД + Storage: Supabase (уже managed)

## Принятые решения

1. **Фото в IndexedDB**: Blob хранятся в Dexie до успешной синхронизации; после sync можно очистить через UI
2. **Client-side UUID**: `crypto.randomUUID()` позволяет создавать записи без обращения к серверу
3. **Порядок синхронизации**: отчёты → фото → финализация (FK-зависимость)
4. **Нет SSR**: полностью клиентский SPA — сервис-воркер кэширует оболочку
5. **JWT 15m + refresh 30d**: баланс между безопасностью и удобством офлайн-работы
6. **Last-write-wins**: в v1 конфликты не разрешаются — записи append-only по своей природе
7. **Dexie v4 schema**: 11 таблиц с compound indexes для sync queue dedup
8. **Supabase PostgreSQL + Cloudflare R2**: заменяют Docker-контейнеры PostgreSQL + MinIO
